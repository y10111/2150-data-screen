# 热轧 2150 线数据化大屏 — AI 模块需求分析与实现方案

> 本文档基于当前仓库 **2150-data-screen** 的实际结构编写，说明：在本项目中 **AI 能做什么**、**如何搭建可对接的 AI 数据化大屏**、以及 **大屏前端如何引用（集成）AI 能力**。可与业务方、算法团队、后端共同评审后迭代。

---

## 一、项目现状：AI 在前端里「长什么样」

### 1.1 界面位置

- **第四行右侧**：`AIAssistant` 组件，标题为「**AI 预警与建议**」，以表格形式展示预警列表（类型、摘要、详情、时间、严重度标识）。
- 与 **设备健康** 并排，同处 `Dashboard.vue` 的 `ai-row-section` 中，外层使用 DataV 边框组件包裹。

### 1.2 数据流（当前实现）

| 环节 | 说明 |
|------|------|
| 请求 | `dashboardApi.getAiAlerts()` → `GET /dashboard/ai/alerts`（见 `src/api/index.js`） |
| Mock | `src/mock/index.js` 将上述路径映射到 `src/mock/data/aiAlertTimeline.json` |
| 聚合 | `Dashboard.vue` 的 `fetchAllData` 中取 **时间轴最后一日** 的 `alerts` 数组，映射为 `aiAlerts` |
| 映射 | `type` 经 `mapAiAlertType` 转为中文类型（预警 / 根因分析 / 预测信息 / 优化建议 等） |
| 展示 | `AIAssistant` 接收 `alerts` prop，支持步进滚动、危险/警告/正常标识 |

**结论**：当前大屏上的「AI」本质是 **结构化预警列表的展示层**；推理、规则、大模型若存在，应在 **后端或独立 AI 服务** 中完成，前端只消费 **已生成好的文案与类型**。

### 1.3 Mock 阶段能力边界（评审用）

**一句话**：现在只有 Mock 时，AI 能干的 =「用这批日数据做 **规则预警** + **统计摘要** + **展示预制话术**」（见 `src/mock/data/aiAlertTimeline.json` 等）；若对外宣传，建议说成 **「智能规则与辅助分析（演示数据）」**，等接入真实 **MES / 时序库** 再谈 **模型与根因**。

| 现阶段可做 | 说明 |
|------------|------|
| 规则型预警 | 基于 `dailyOperations.json` 等字段写阈值/环比（热装率、停机、库存等），生成或筛选告警条目。 |
| 统计与对比 | 环比、累计欠产、分设备类型描述性统计；仅辅助读数，不构成因果结论。 |
| 预制话术展示 | 大屏「AI 预警与建议」当前主要展示 **已写入 Mock 的 title/message**，可改文案与类型，但内容仍来自数据文件或后续规则引擎。 |
| 可选 LLM 润色 | 若接大模型，可把「当日 KPI JSON」生成摘要——**输入仍来自 Mock/演示接口**，适合演示，不代表现场实测。 |

| 现阶段不宜对外承诺 | 说明 |
|--------------------|------|
| 可靠长期预测 | Mock 仅约二十余日日级点，无小时级与外部变量，预测只能作示意。 |
| 工业级根因诊断 | 无工单、无点位级工艺参数，仅有 `device_type` 等标签，只能做「关联提示」。 |
| 在线学习与越用越准 | 无持续真实数据回流则无法成立。 |

**你需要做的**：评审时采用上表表述即可；落地真实能力时，让后端接入生产数据并替换 Mock，再与算法团队约定模型与接口。

### 1.4 已接入：本地 `screen_ai_service`（Qwen）

- 开发环境下 **`/api/dashboard/ai/alerts` 不再走 Mockjs**，由 Vue 开发服务器 **代理至** `http://127.0.0.1:5001`（见 `vue.config.js`、`.env.development` 中 `VUE_APP_SCREEN_AI_URL`）。
- 仓库根目录 **`npm run ai-server`** 或进入 `screen_ai_service` 执行 **`python app.py`** 启动 LLM 服务；未启动时大屏其余模块仍可用，AI 预警区为空并在控制台警告。
- 详见根目录 `README.md` 与 `screen_ai_service/README.md`。

## 二、在本项目中，AI「能做什么」（能力边界）

结合已有 KPI、趋势、质量、设备、库存等数据（见 `dailyOperations.json`、`qualityDefects.json` 等），AI 或智能分析可在以下方向创造价值：

### 2.1 适合做的（与数据大屏强相关）

| 能力 | 说明 | 与本项目数据的结合方式 |
|------|------|------------------------|
| **异常检测与分级预警** | 对产量、成材率、热装率、煤气单耗、质量封锁、停机、库存等做规则或模型判断，输出「是否异常 + 级别」 | 日级序列 + 阈值/统计（如 3σ、环比） |
| **自然语言式摘要** | 将当日关键指标压缩成一句话摘要、列表说明（当前表格里的 `title` / `message` 即此类） | 可由 **模板规则** 或 **LLM** 根据当日 KPI JSON 生成 |
| **根因提示（辅助）** | 在异常发生时，列出「可能关联因素」（设备类型、停机时长、热装偏低等） | 需历史样本；初期可用 **规则树**，后期用 **可解释模型** |
| **短期预测** | 未来 1～3 日产量、库存区间等 | ARIMA / Prophet / 简单滑动预测均可；结果可写入「预测信息」类告警 |
| **优化建议文案** | 如「建议协调炼钢」「关注卷取」等可执行提示 | 规则 + 行业知识库；或用 LLM 在 **受控 prompt** 下生成 |
| **设备健康联动** | 健康评分偏低时，在 AI 区推送关联解释（与 `DeviceHealthStrip` 数据同源扩展） | 同一套日数据上增加评分解释接口 |

### 2.2 需要额外条件才能做好的

- **强因果根因**：需要标注数据、工单、MES 明细，单靠日级聚合往往只能做「关联提示」而非定论。
- **实时秒级 AI**：当前大屏以 **日批 / 定时拉取** 为主，若要做产线级实时，需流式数据管道与单独的计算层。

### 2.3 不适合单靠「大屏前端」做的

- 模型训练、重推理、长上下文对话：**不应** 放在浏览器内；应通过 **API** 由服务端或模型服务完成。

## 三、如何搭建「可对接 AI」的数据化大屏（架构建议）

### 3.1 分层架构（推荐）

```
┌─────────────────────────────────────────────────────────┐
│  展示层（本仓库 Vue 大屏）                                  │
│  KPI / ECharts / AI 预警表格 / 设备健康 …                   │
└───────────────────────────┬─────────────────────────────┘
                            │ HTTPS / WebSocket（可选）
┌───────────────────────────▼─────────────────────────────┐
│  应用接口层（BFF 或后端聚合）                               │
│  /dashboard/kpi / trend / … /dashboard/ai/alerts         │
└───────────────────────────┬─────────────────────────────┘
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
   ┌──────────┐      ┌─────────────┐    ┌──────────────┐
   │ 业务库    │      │ 规则引擎     │    │ AI 服务       │
   │ 时序/报表 │      │ 阈值/环比    │    │ Python 模型   │
   └──────────┘      └─────────────┘    │ 或 LLM API    │
                                        └──────────────┘
```

- **展示层**：只负责 **读接口、渲染、轮询/刷新**（本项目已在 `fetchAllData` 中统一拉数）。
- **AI 服务**：可以是 **纯规则**、**Python 批处理任务写回数据库**、或 **调用云厂商大模型**；对前端统一暴露为 **REST JSON**。

### 3.2 与本项目对齐的最小落地方式

1. **保持前端数据结构不变**：继续提供 `GET /dashboard/ai/alerts`，返回体与 `aiAlertTimeline.json` 中单日结构兼容（`date` + `alerts[]`）。
2. **替换 Mock**：生产环境关闭 Mock，指向真实后端；后端从规则引擎或 AI 服务组装 `alerts`。
3. **扩展字段（可选）**：在 `alerts` 项中增加 `suggestion`、`related_metric`、`confidence`，前端表格可增加列或在详情中展示（需改 `AIAssistant.vue`）。

## 四、本项目如何「引用」AI（集成步骤）

### 4.1 前端已预留的引用点

1. **API**：`src/api/index.js` 中 `getAiAlerts`。
2. **页面注入**：`Dashboard.vue` 将 `aiAlerts` 传入 `<AIAssistant :alerts="aiAlerts" />`。
3. **类型映射**：`mapAiAlertType` 将后端 `info` / `warning` / `alert` / `suggestion` 等映射为中文类型，并与 `SeverityStatusIcon` 的 **危险 / 警告 / 正常** 策略一致。

### 4.2 接入真实后端时建议步骤

1. 实现服务端 `GET /api/dashboard/ai/alerts`（或保持现有 `request` 的 baseURL 规则）。
2. 返回格式建议：

```json
{
  "code": 200,
  "data": [
    {
      "date": "2026-03-28",
      "alerts": [
        {
          "id": 1,
          "type": "warning",
          "title": "热装率偏低",
          "message": "热装率 42.6%，低于目标 55%，建议……"
        }
      ]
    }
  ]
}
```

3. 若只返回「最新一天」，仍可包成 **单元素数组**，与现有 `aiAlertsData[aiAlertsData.length - 1]` 逻辑兼容。
4. **定时刷新**：在 `Dashboard.vue` 的 `mounted` 后增加 `setInterval(fetchAllData, 周期)` 或单独 `refreshAiOnly()`，按大屏需求设为 1～5 分钟（注意性能与后端压力）。

### 4.3 若引入大模型（LLM）

- **推荐模式**：后端调用 LLM，将「当日 KPI JSON + 业务约束 prompt」生成 `title` / `message`，**不要** 把生产网数据库直连公网模型。
- **前端**：仍只消费 `alerts` 字符串；可选增加「生成时间」「模型版本」字段便于审计。

## 五、AI 功能设计与算法映射（延续原方案精华）

以下可作为 **算法团队** 的需求说明；实现可先从 **规则模拟** 再过渡到 **模型**。

| 功能 | 思路简述 | 难度 |
|------|----------|------|
| 实时异常检测 | 动态阈值、3σ、环比阈值 | 低 |
| 根因分析 | 关联规则、决策树、专家规则 | 中 |
| 趋势预测 | ARIMA、Prophet、XGBoost | 中 |
| 优化建议 | 回归、线性规划、规则库 | 中～高 |
| 自然语言问答 | 关键词路由或 LLM | 中 |

**开发阶段**：可用规则在服务端生成 `alerts`，与当前 Mock 形态一致，保证大屏联调顺畅。

## 六、建议的后端 API 扩展（便于真实 AI 引擎）

在仅读 `alerts` 的基础上，后续可增量增加（不影响现有表格最小展示）：

| 接口 | 用途 |
|------|------|
| `POST /api/ai/anomaly` | 单指标异常判定（供调试或别的系统） |
| `POST /api/ai/root_cause` | 根因结构化结果 |
| `POST /api/ai/forecast` | 预测序列 |
| `POST /api/ai/suggestion` | 优化建议列表 |
| `POST /api/ai/chat` | 对话（若大屏增加侧栏问答） |

前端对这类接口可采用 **按需调用** 或与 BFF **合并进** `ai/alerts` 的聚合结果，避免大屏发起过多请求。

## 七、可视化与交互（与本项目一致的部分）

- **第四行布局**：设备健康 + AI 预警并排；已使用 `dv-border-box` 与统一样式表格（见 `Dashboard.vue`、`DeviceHealthStrip.vue`、`AIAssistant.vue`）。
- **严重度**：`SeverityStatusIcon` 区分危险 / 警告 / 正常，与类型映射一致。
- **滚动**：AI 与设备健康均采用 **步进停留滚动**，避免连续跑马灯干扰阅读。

后续若增加「点击某条预警高亮相关图表」，可在 `Dashboard` 层增加 **事件总线或 provide/inject** 联动 ECharts 实例。

## 十、总结

- 本项目的 **AI 模块在前端** 体现为 **结构化预警与建议列表**，通过 **`/dashboard/ai/alerts`** 与 **`AIAssistant`** 引用；**智能逻辑应后置** 到规则引擎或独立 AI 服务。
- **搭建 AI 数据化大屏** 的要点是：**展示层薄、接口契约稳定、数据与推理分层**；本仓库已具备 **接入真实 AI 结果的最小闭环**。
- 按 **阶段一 → 四** 演进，可在保证演示与生产可用的前提下，逐步增强 **检测、预测、根因、LLM 摘要** 等能力。
